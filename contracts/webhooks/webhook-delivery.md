# Contract 21 — Outbound Webhook Delivery ⚡

> **Status:** ⚡ First Cycle. Normative. How external customers consume platform events over
> HTTPS webhooks.

External customers consume platform events (billing, tenant lifecycle, ingestion completion,
guardrail violation, tool-invocation completion). **They cannot read CypherX's Kafka. They cannot
poll efficiently.** They need **HTTPS webhooks**.

---

## 1. Subscription model

```sql
CREATE TABLE platform.webhook_subscriptions (
  subscription_id   UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  tenant_id         UUID NOT NULL,
  url               TEXT NOT NULL,
  event_types       TEXT[] NOT NULL,            -- ['cypherx.llms.usage.recorded', 'cypherx.tenant.plan_changed', ...]
  signing_secret    TEXT NOT NULL,              -- HMAC-SHA256 key, 32 bytes base64, generated by platform, shown ONCE
  status            TEXT NOT NULL DEFAULT 'active' CHECK (status IN ('active', 'paused', 'disabled')),
  failure_count     INTEGER NOT NULL DEFAULT 0,
  last_success_at   TIMESTAMPTZ,
  created_at        TIMESTAMPTZ NOT NULL DEFAULT NOW(),
  api_version       TEXT NOT NULL DEFAULT 'v1'
);
ALTER TABLE platform.webhook_subscriptions ENABLE ROW LEVEL SECURITY;
CREATE POLICY p_webhook_subs ON platform.webhook_subscriptions
  USING (tenant_id = current_setting('app.tenant_id')::uuid);
```

### Column rules

| Column | Rule |
|--------|------|
| `subscription_id` | UUID PK, server-generated. |
| `tenant_id` | UUID NOT NULL. Tenant-scoped; RLS keyed on `app.tenant_id` (Contract 13 §4). |
| `url` | Receiver endpoint (HTTPS). |
| `event_types` | `TEXT[]` of subscribed event types (e.g. `cypherx.llms.usage.recorded`, `cypherx.tenant.plan_changed`). |
| `signing_secret` | HMAC-SHA256 key, **32 bytes base64**, **generated by the platform**, **shown ONCE**. |
| `status` | One of `active` \| `paused` \| `disabled` (CHECK-constrained); default `active`. |
| `failure_count` | INTEGER, default 0. Reset to 0 on success; drives auto-pause (§4). |
| `last_success_at` | TIMESTAMPTZ of the last 2xx delivery. |
| `created_at` | TIMESTAMPTZ, default NOW(). |
| `api_version` | TEXT, default `v1`. |

## 2. Delivery shape

```
POST <subscription.url>
Content-Type: application/json
User-Agent: cypherx-webhook/1.0
X-CypherX-Event:      <event_type>
X-CypherX-Event-ID:   <uuid — unique per delivery attempt>
X-CypherX-Timestamp:  <unix epoch seconds>
X-CypherX-Signature:  v1=<hex(HMAC_SHA256(signing_secret, timestamp + '.' + body))>
X-CypherX-Attempt:    <integer, 1-indexed>
X-CypherX-Delivery-ID:<uuid — same across retries of same event>

Body: the full Kafka event envelope (Contract 5) for that event.
```

### Header semantics

| Header | Meaning |
|--------|---------|
| `Content-Type` | Always `application/json`. |
| `User-Agent` | Always `cypherx-webhook/1.0`. |
| `X-CypherX-Event` | The `<event_type>`. |
| `X-CypherX-Event-ID` | UUID **unique per delivery attempt** (changes on every retry). |
| `X-CypherX-Timestamp` | Unix epoch **seconds** — signed and replay-checked. |
| `X-CypherX-Signature` | `v1=<hex(HMAC_SHA256(signing_secret, timestamp + '.' + body))>`. |
| `X-CypherX-Attempt` | Integer, **1-indexed** attempt number. |
| `X-CypherX-Delivery-ID` | UUID **stable across all retries of the same event** — the idempotency key. |

**Body** = the **full Kafka event envelope (Contract 5)** for that event.

---

## 3. Signature verification by the receiver

1. **Reject if `|now - X-CypherX-Timestamp| > 300` seconds** (replay protection — 300s window).
2. Recompute `expected = hex(HMAC_SHA256(secret, timestamp + '.' + body))`.
3. Compare with **`constant_time_equal`** to the value after `v1=`.

> The signed string is **`timestamp + '.' + body`** — i.e. the `X-CypherX-Timestamp` value, a literal
> `.`, then the raw request body. Use the raw bytes, not a re-serialized copy.

---

## 4. Delivery semantics

- **At-least-once.** **Receivers MUST be idempotent on `X-CypherX-Delivery-ID`** (the same logical
  event may arrive more than once across retries).
- **Retries — exponential backoff, exactly 10 attempts over ~3 days:**

  | Attempt | Delay before this attempt |
  |---------|---------------------------|
  | 1 | (immediate) |
  | 2 | 1s |
  | 3 | 5s |
  | 4 | 30s |
  | 5 | 5m |
  | 6 | 30m |
  | 7 | 2h |
  | 8 | 12h |
  | 9 | 24h |
  | 10 | 24h |
  | (final) | 24h |

  Backoff schedule: **1s, 5s, 30s, 5m, 30m, 2h, 12h, 24h, 24h, 24h — 10 attempts over ~3 days.**
- **2xx response → delivery succeeded.** Subscription `failure_count` **resets to 0** (and
  `last_success_at` updates).
- **Non-2xx or timeout (>10s):** increment `failure_count`. After **10 consecutive failures**,
  `status='paused'` and an **alert email is sent to the subscription's owner**. Manual
  **`POST /v1/webhooks/{id}/resume`** reactivates.
- **4xx with a body containing `{"retry": false}`** → **drop immediately** (the receiver explicitly
  rejected the delivery; no further retries).

---

## 5. Per-tenant rate limits

- Governed by **`webhook_deliveries_per_min`** from Contract 19 quotas.
- **Defaults:** `1000/min` for `pro`, `100/min` for `free`.

---

## 6. Replay endpoint

- **`POST /v1/webhooks/{id}/replay { event_id }`** → re-delivers a **stored event up to 30 days old**.
- Purpose: **subscriber-side bug recovery without losing data.**
